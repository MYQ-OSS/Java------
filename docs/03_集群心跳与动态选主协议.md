# 03_集群心跳与动态选主协议

> **项目名称**：多线程C/S架构NoSQL数据库系统
> **设计人**：马奕钦 (组长)
> **版本**：v1.0.0
> **更新时间**：2026-07-11

---

> **架构迭代重要声明**：在集群通信中，项目将原定的自定义二进制包 (OpCode = 0x07) 直接升级为基于标准 Redis 协议的集群控制指令，包括 `CLUSTER_ELECT` (投票请求/回应)、`CLUSTER_HEARTBEAT` (单向任期心跳)、`CLUSTER_REPLICATE` (指令广播复制) 和 `CLUSTER_SYNC` (数据快照全量同步)。主从数据同步过程中，采用特殊的 `\u0001` 分隔符对数据实体进行隔离传输，完美消除了由于 Value 中包含空格等特殊字符引起的分隔冲突，确保了在极端网络干扰下集群数据的强一致性。同时保留 3 节点动态选主状态机，具备极高的鲁棒性。

集群模式默认部署 3 个节点，支持：
1. **Master (主节点)**：唯一允许接收客户端“写操作”（PUT/DELETE）的节点。负责将变更日志广播同步到其他从节点。
2. **Slave (从节点)**：仅允许接收客户端“读操作”（GET/SCAN）并作为 Master 数据的热备份。
3. **读写分离路由**：客户端 SDK 负责管理集群连接池，写请求路由到 Master，读请求在 3 个节点间实现负载均衡。

## 二、 动态角色选举机制 (基于 Raft 简化版)

为了解决单点故障，系统支持在启动时通过网络广播与心跳投票，实现 **动态角色选举（组队必做项）**。

### 1. 节点状态机转换 (State Machine)
每个集群节点在生命周期中可能处于以下三种状态之一：
- **Follower (跟随者)**：只接收主节点的同步与心跳。如果心跳超时，则升级为 Candidate。
- **Candidate (候选者)**：发起一轮选举投票，向其他节点索要选票。若获得半数以上票数，则升级为 Leader。
- **Leader (领导者/Master)**：定时向其他节点发送心跳。接收并处理所有客户端写入。

```
     [ 启动 ]
        │
        ▼
   ┌──────────┐      心跳超时      ┌───────────┐
   │ Follower ├─────────────────>│ Candidate │
   └────▲─────┘                  └─────┬─────┘
        │                              │  获得超半数选票
        │          收到新Leader心跳    │  (>= 2票)
        └──────────────────────────────┼──────────┐
                                       ▼          ▼
                                  ┌─────────┐ ┌────────┐
                                  │ Leader  │ │ Candidate│ (落选)
                                  └─────────┘ └────────┘
```

### 2. 选举协议包设计 (OpCode = 0x07)
选举报文包含以下信息：
- `term` (当前任期号)：递增的整数，用于标识选举周期。
- `candidateId` (候选者节点 ID)：如 `node_1`、`node_2`、`node_3`。
- `lastLogIndex` (本地最后一条 WAL 日志索引)：用于评估谁的数据最新，数据越新越有资格成为主。

### 3. 选举算法步骤
1. **心跳超时触发**：Follower 在配置的选举超时时间（随机设置在 150ms ~ 300ms 之间，防止选票分裂）内未收到 Leader 心跳，将本地 `term` 加 1，状态转为 Candidate，并给自己投一票，同时向其他节点发送拉票请求（`RequestVote`）。
2. **选票裁决**：其他节点收到拉票请求后，如果请求的 `term` 大于自身，且请求者数据比自己新（比较 `lastLogIndex`），则投赞成票并重置心跳定时器。
3. **晋升 Leader**：Candidate 收集到 2 张或以上的选票后，正式晋升为 Leader，并开始向其他节点发送 Leader 心跳（`AppendEntries`）。

## 三、 主从数据同步机制 (Replication)

### 1. 同步数据流
1. 客户端向 Master 发送 `PUT key=value`。
2. Master 接收后：
   - 写入本地 WAL 日志并更新本地内存。
   - 异步将变更操作封装为 `ReplicationPacket`，发送给所有的 Slave 节点。
3. Slave 节点接收到变更请求后，写入本地 WAL 与内存，并向 Master 发送响应 ACK。

```
Client ----(1) Write----> Master ----(2) Replication----> Slave 1 & Slave 2
                            |                                |
  |<---(4) Success Ack------+<-----------(3) ACK ----------+
```

### 2. 故障恢复与增量同步 (Incremental Sync)
若某个 Slave 节点短暂断网后重连，其数据必然落后于 Master：
- Slave 启动时将向 Master 报告其本地最新的 `lastLogIndex`。
- Master 检查发现落后，则在 WAL 历史记录中找到对应 Index 之后的全部变更流，增量同步给该 Slave，直到 Slave 追赶上主节点的数据进度。

## 四、 阶段交付物与验证机制 (马奕钦)
1. **交付类**：`ClusterNode`（表示节点实体）、`RaftConsensusModule`（状态机核心选举逻辑）、`HeartbeatTimer`（心跳计时器）。
2. **验证方法**：
   - 本地启动 3 个进程（占用不同端口如 8081, 8082, 8083），控制台输出日志可见 `Node 1 elected as Leader`。
   - 手动关闭（Kill）Node 1，控制台输出 `Leader disconnected. Initiating new election...`，几百毫秒内 Node 2 或 Node 3 打印 `Node X promoted to Leader`，此时数据库依旧能支持读写。
