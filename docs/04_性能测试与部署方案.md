## 一、 系统部署与运行方案

项目使用 **JDK 21** 作为开发和运行环境。不需要引入复杂的外部组件，纯 Java 原生编译。

### 1. 单机模式运行部署
编译出服务端 `server.jar` 和命令行客户端 `cli.jar`。
- **启动 Server**：
  ```bash
  java -jar server.jar --port 8080 --mode standalone --data-dir ./data
  ```
- **启动 CLI 客户端**：
  ```bash
  java -jar cli.jar --host 127.0.0.1 --port 8080
  ```

### 2. 集群模式运行部署
集群采用 3 节点分布式部署。
- **启动 Node 1 (Master/Follower 候选)**：
  ```bash
  java -jar server.jar --port 8081 --mode cluster --id node_1 --peers 127.0.0.1:8082,127.0.0.1:8083 --data-dir ./data/n1
  ```
- **启动 Node 2**：
  ```bash
  java -jar server.jar --port 8082 --mode cluster --id node_2 --peers 127.0.0.1:8081,127.0.0.1:8083 --data-dir ./data/n2
  ```
- **启动 Node 3**：
  ```bash
  java -jar server.jar --port 8083 --mode cluster --id node_3 --peers 127.0.0.1:8081,127.0.0.1:8082 --data-dir ./data/n3
  ```

## 二、 性能与压力测试方案 (Benchmark)

为满足指导书中“系统性能测试”的要求，我们需要使用多线程对数据库进行吞吐量与耗时测试。

### 1. 测试工具与指标定义
我们将编写一个专门的性能测试程序 `BenchmarkTool.java`：
- **测试环境**：Windows 11, Core i7/Ryzen 7 处理器, 16GB RAM, SSD。
- **主要指标**：
  - **QPS (Queries Per Second)**：每秒完成的读/写请求次数。
  - **Latency (时延)**：单次操作的平均时延、99% 时延 (P99)。
  - **并发客户端数**：10, 50, 100 线程并发。

### 2. 测试场景设计
- **场景 A：纯内存读写测试**（缓存全命中，测试内存引擎极限 QPS）
- **场景 B：混合读写落盘测试**（读写比 8:2，开启持久化与 WAL，测试持久化下的性能表现）
- **场景 C：大文件 Rotate 与多线程压缩开销测试**（不间断写入大数据，触发 Rotate，监控文件压缩时对读写 QPS 的波动影响）

## 三、 测试用例表格设计 (Test Cases)

| 用例 ID | 测试模块 | 测试输入描述 | 预期结果 | 实际执行状态 |
| :--- | :--- | :--- | :--- | :---: |
| **TC-01** | 单机 CRUD | PUT 插入 KV，GET 读取，DELETE 删除，批量写入 100 条。 | 数据读写一致，批量操作正常完成。 | 🟡 待测 |
| **TC-02** | 自动 Rotate | 持续写入数据，直到活跃数据文件 `active.db` 达到 10MB。 | 触发切分，异步生成 `.db.gz` 压缩文件，磁盘空间正常开销。 | 🟡 待测 |

| **TC-06** | 集群主从同步 | 向新 Master 节点写入数据。 | 数据自动广播，在 2 个 Slave 节点同步落盘并能读出。 | 🟡 待测 |
