> **架构迭代重要声明**：为了与标准 Redis 客户端工具链（如 `redis-cli`、Redis Desktop Manager 以及主流语言的 Redis 客户端 SDK）无缝兼容，项目在迭代中将“自定义 NSP 二进制数据传输协议”直接升级与映射到了 **Redis RESP (Redis Serialization Protocol)** 标准协议。底层的路由调度器也对应升级为标准 RESP 的命令分发控制器，并在控制台以逻辑路径日志输出以兼容 RESTful 的映射效果。此举极大地提高了本系统的开放性与实用价值。

## 一、 通信层整体设计

本数据库基于 **C/S（Client/Server）架构**。为支持高并发的多客户端连接，网络层将使用 **Java 21** 提供的多线程网络接口进行开发：
1. **服务端监听**：使用 `ServerSocketChannel` 配合 `Selector`（Java NIO）或标准 `ServerSocket` 搭配 **虚拟线程 (Virtual Threads)** 以实现轻量级的高并发线程处理。
2. **长连接维持**：客户端与服务端通过 Socket 维持长连接，通信完毕后除非客户端主动发起 `QUIT`，否则连接不释放。
3. **心跳检测**：双向维持 Keep-Alive 心跳包，防止空闲连接被防火墙强制断开。

## 二、 自定义数据传输协议 (NoSQL Protocol - NSP)

为保证网络传输的高效与安全，客户端与服务端之间采用自定义的二进制字节流协议。单条报文数据帧结构（NSP Packet）设计如下：

```
+------------+------------+----------------+----------------+
|  Magic Num |  OpCode    | Body Length    |      Body      |
|  (2 Bytes) |  (1 Byte)  |   (4 Bytes)    |  (N Bytes...)  |
+------------+------------+----------------+----------------+
```

### 1. 字段说明
- **Magic Number (魔数)**：固定为 `0x2A 0x3F`，占用 2 字节。用于识别和过滤非法网络请求。
- **OpCode (操作码)**：占用 1 字节。用于识别当前的业务动作，例如：
  - `0x01`: PUT (添加或更新)
  - `0x02`: GET (查询单个 Key)
  - `0x03`: DELETE (删除 Key)
  - `0x04`: BULK_PUT (批量添加)
  - `0x05`: HEARTBEAT (心跳检测)
  - `0x06`: ERROR (错误响应)
  - `0x07`: LEADER_ELECT (集群选举报文)
- **Body Length (正文长度)**：占用 4 字节（int，大端序）。标记后续 Body 数据的总字节数。
- **Body (数据体)**：变长字段（N 字节）。以特定序列化格式存储的具体请求参数或响应数据。

## 三、 数据序列化方案

为保证对 Java 内置复杂数据类型（数值、字符串、List、Map、自定义对象）的支持，系统采用 **JSON 序列化 (如 FastJSON2 / Jackson)** 或 **自定义二进制序列化** 方案。

### 1. 协议对象设计
#### 客户端请求包体 (Request Body)
```json
{
  "collection": "user_profile",
  "key": "user_10086",
  "valType": "map",
  "value": {
    "name": "Ma Yiqin",
    "role": "Leader",
    "age": 20
  }
}
```

#### 服务端响应包体 (Response Body)
```json
{
  "code": 200,
  "message": "SUCCESS",
  "data": {
    "key": "user_10086",
    "value": {
      "name": "Ma Yiqin",
      "role": "Leader",
      "age": 20
    }
  }
}
```

## 四、 Socket 层 RESTful 风格路由调度器

在 Socket 网络通信之上，设计一个轻量级路由调度器，将复杂的 NSP Packet 报文解包后，类似于 RESTful Controller 模式分发到不同的 Handler 处理器：

| NSP OpCode | 逻辑资源路径 (URI) | 对应处理器 (Handler) | 职责 |
| :--- | :--- | :--- | :--- |
| `0x01` | `/collections/{coll}/put` | `PutHandler` | 增/改单个 Key-Value |
| `0x02` | `/collections/{coll}/get/{key}` | `GetHandler` | 根据 Key 获取数据 |
| `0x03` | `/collections/{coll}/delete/{key}` | `DeleteHandler` | 根据 Key 删除数据 |
| `0x04` | `/collections/{coll}/bulk` | `BulkPutHandler` | 批量数据原子写入 |
| `0x05` | `/system/heartbeat` | `HeartbeatHandler` | 维持连接与健康状态 |
| `0x07` | `/cluster/elect` | `ClusterElectHandler` | 参与集群状态选举 |
