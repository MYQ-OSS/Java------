> **项目名称**：多线程C/S架构NoSQL数据库系统
> **设计人**：马奕钦 & 黎展宏
> **版本**：v1.0.0
> **更新时间**：2026-07-11

---

> **架构迭代重要声明**：在落盘与持久化实现中，项目将原定的 `active.db` 和 `wal.log` 分离的二进制格式直接升华与整合为标准的 **Redis AOF (Append Only File) 持久化格式**。AOF 既充当了高安全性的 WAL 追加日志，又能在 Rotate 时实现多线程压缩重写。单机引擎中更独创实现了基于 `LruCache`（热数据）与磁盘文件（冷数据）的冷热分离机制，并引入内存中的 **O(1) 磁盘偏移量索引 (Disk Offset Index)** 以极大地提升大数量下的检索性能。

## 一、 内存数据模型

数据库内部通过多层 Map 来组织 Collection 与 Key-Value 的层级关系。
- **全局管理**：`ConcurrentHashMap<String, CollectionEngine>` 用于存储 Collection 映射。
- **Collection 模型**：每一个 Collection 内维护一个 `ConcurrentHashMap<String, DbValue>`。
- **数据结构支持**：`DbValue` 对象定义了具体的值。值类型包含：
  - `String`（基础字符型）
  - `Number`（整型、浮点型）
  - `List`（列表类型）
  - `Set`（集合类型）
  - `Map`（哈希映射类型）

---

## 二、 持久化与文件 Rotate 机制

为实现数据永久保存与灾难重建，系统采用类似于 Append-Only File (AOF) 配合 LSM-Tree 的段（Segment）存储设计。

### 1. 数据落盘存储格式 (Segment File)
数据写入内存后，将以二进制或者 JSON 行形式追加写入当前的活跃数据文件 `active.db`：
```
[TIMESTAMP]|[OP_TYPE]|[COLLECTION]|[KEY]|[VALUE_TYPE]|[VALUE_CONTENT]\n
```

### 2. 自动 Rotate（分卷）流程
为避免单个数据文件无限变大，设计 **自动 Rotate 机制**：
1. **大小阈值监控**：在每次写入后检查 `active.db` 的文件大小（如设置为 10MB）。
2. **切分重命名**：若超出阈值，将 `active.db` 锁定并重命名为 `segment_timestamp.db`。同时开启一个新的 `active.db` 接收新写入。
3. **多线程异步压缩**：
   - 触发一个后台线程池（使用 `ExecutorService`）。
   - 异步对重命名后的 `segment_timestamp.db` 文件进行 **GZIP 压缩**，生成 `segment_timestamp.db.gz`，并物理删除原 `.db` 文件。

## 三、 WAL (Write-Ahead Log) 预写日志与异常恢复

为解决高并发写操作在内存与磁盘之间的非一致性，提供断电数据防丢失保障：

```
Client ----(1) Write WAL----> WAL File (Append Only & fsync)
  |
  +--------(2) Update Mem----> Memory Engine ----(3) Ack to Client
```

### 1. 预写日志机制
在写入操作更新内存中 `ConcurrentHashMap` 之前，必须先将操作记录追加写入 `wal.log` 日志文件，并强制刷盘（`FileChannel.force(true)`）。

### 2. 异常回放恢复
当数据库 Server 重启（无论是正常重启还是断电崩溃后重启）：
1. 扫描配置的日志目录，寻找 `wal.log` 文件。
2. 逐行读取 `wal.log` 记录的操作指令，在内存中重新执行一遍，完成数据回放恢复。
3. 清空 `wal.log` 并重新开启日志。

## 四、 查询优化：缓存与索引设计

### 1. 数据双级缓存 (Cache System)
为减少磁盘 IO：
- **一级缓存（L1）**：使用基于内存的 LRU（Least Recently Used）缓存淘汰策略，使用 `LinkedHashMap` 实现。
- **二级缓存（L2）**：对于频繁读取的 Collections 建立局部热数据缓存。
- **一致性保证**：当发生 `PUT` / `DELETE` 写操作时，主动失效对应 Key 缓存（Cache Eviction），防止读取脏数据。

### 2. 磁盘文件索引 (File Indexing)
- **Hash 索引**：内存中维护一个 `ConcurrentHashMap<String, Long>`，Key 为数据键，Value 为该数据在持久化磁盘文件中的物理字节偏移量 (Offset)。
- **查询加速**：当缓存未命中时，利用 `RandomAccessFile` 的 `seek(offset)` 直接定位读取磁盘中的目标数据，将磁盘查询效率提升至 O(1)。
